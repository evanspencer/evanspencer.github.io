How I Built a Fintech Analytics Dashboard in One Session with Claude Code

May 9, 2026 7 min read Evan Spencer

Every analyst who works with payment data runs into the same wall. You can see something interesting in SQL — a pattern in how companies spend, a clustering of transaction behavior, a product signal hiding in merchant categories — but turning that into something your stakeholders can actually explore requires a whole other skillset. You either wait for eng time, build something janky in Excel, or scope the insight down until it fits in a slide deck.

I’ve been working with virtual card transaction data at BILL for a while now. I’m comfortable in SQL and I write Python when I need to, but building a real interactive dashboard — with proper data generation, meaningful categorization logic, and a visualization that actually tells a story — felt like a full sprint of eng work. Last week I did it in an afternoon, pair programming with Claude Code.

Here’s what I built, how the session actually went, and what I think it means for analysts who want to ship things.

The business question I’d been sitting on

Virtual card transactions carry a Merchant Category Code — a four-digit MCC assigned by the card network. MCC 5045 is computers and peripherals. MCC 7372 is prepackaged software. MCC 7311 is advertising agencies. There are hundreds of these codes, and for B2B fintech, they’re the canonical taxonomy of what companies actually spend money on.

The question I kept returning to: when a company uses a virtual card product, which merchant categories cluster together? If a company uses cards heavily for SaaS subscriptions, does that predict anything about their advertising spend? Their travel patterns? This is a product affinity question — do certain spend behaviors travel together, and if so, what does that tell us about which product segments we’re actually serving?

Answering it properly meant building something interactive. I wanted to hand a PM an artifact they could filter and explore, not a static chart in a slide.

Starting with synthetic data

The first design constraint was obvious: real virtual card transaction data lives behind access controls, legal agreements, and infrastructure I can’t publish to a public GitHub repo. So I needed synthetic data that was structurally realistic — not random noise, but something that preserved the statistical properties that make the analysis meaningful.

This is where the session with Claude Code started. I described the schema I needed: transactions with merchant names, MCC codes, amounts, card product types, and company-level attributes like size tier and industry vertical. Then I described what “structurally realistic” actually meant for B2B payment data.

The key insight we worked through together: real B2B spend has fat tails. A few merchants capture most of the volume. Software subscriptions are small and frequent. Vendor payments are large and rare. A uniform distribution across amounts would produce a dashboard that looks clean but lies about the underlying reality.

We landed on lognormal distributions parameterized by MCC category, calibrated to match the rough shape of what I know from working with real data: 

# Amount distributions vary significantly by spend category
MCC_AMOUNT_PROFILES = {
    "software":    {"dist": "lognormal", "mu": 4.2, "sigma": 1.1},   # ~$60–$5k
    "advertising": {"dist": "lognormal", "mu": 6.8, "sigma": 1.4},   # ~$500–$50k
    "travel":      {"dist": "lognormal", "mu": 5.1, "sigma": 1.6},   # ~$150–$8k
    "utilities":   {"dist": "lognormal", "mu": 4.8, "sigma": 0.8},   # ~$120–$800
}

Claude Code wrote the Faker-based generation scaffolding quickly once I had the statistical shape right. What I brought to that exchange was knowing which distributions to use and why. What I didn’t need to bring: the Faker API, the NumPy sampling calls, or the pandas DataFrame construction.

The categorization layer

Raw MCC codes aren’t useful for analysis — there are too many of them (hundreds of four-digit codes) and they’re granular in ways that don’t map cleanly to analytical questions. The spend categorizer needed a bucketing layer that collapsed MCCs into a smaller set of meaningful categories.

This was the first place where the judgment calls were genuinely mine. Should “computers and peripherals” (MCC 5045) live in the same bucket as “prepackaged software” (MCC 7372)? Analytically, yes — both are tech spend, both tend to co-occur with SaaS-heavy companies. Should “airlines” (MCC 3000–3299) and “hotels” (MCC 7011) merge into a single “travel” category? Probably, unless you care about the company-level distinction.

I landed on 12 top-level categories: software, advertising, travel, utilities, professional_services, office_supplies, food_entertainment, logistics, hardware, telecom, financial_services, and other. The final mapping dictionary was something I drafted based on domain knowledge, then had Claude Code clean up and validate for completeness. 

def categorize_mcc(mcc_code: int) -> str:
    """Map a four-digit MCC to a top-level spend category."""
    if 5045 == mcc_code or 7372 == mcc_code:
        return "software"
    elif mcc_code in ADVERTISING_MCCS:
        return "advertising"
    elif 3000 <= mcc_code <= 3299 or mcc_code == 7011:
        return "travel"
        # ... remaining categories
        return "other"

Building the affinity visualization

The centerpiece of the dashboard is a bubble chart showing merchant category spend patterns broken out by card product type. The encoding:

  • X axis — average transaction frequency per company per month
  • Y axis — median transaction amount (log scale)
  • Bubble size — number of companies using this category
  • Color — card product type (SaaS-focused, expense management, AP automation)

The goal is to reveal structure: do high-frequency, low-amount transactions cluster in one product type? Do infrequent, high-value transactions indicate a different use case? If the product types genuinely serve different spend behaviors, the chart should show them separating in the space.

What I learned building this: the hardest part wasn’t the Plotly code. It was figuring out what affinity actually meant as something measurable. High co-occurrence between categories? Correlation in spend share across companies? I went with a simpler proxy — Pearson correlation between category spend-share vectors across companies — which isn’t perfect but is interpretable to a PM without a stats background.

Claude Code wrote the Plotly figure code quickly once I had the data shape nailed down. What I had to bring to that exchange: knowing that the color encoding should be the product dimension (not the MCC), knowing what fields needed to be in the hover text, and knowing that the Y axis needed a log scale because spend amounts span two orders of magnitude.

What Claude Code did and didn’t do

I want to be honest about this, because the marketing version of this story (“AI wrote my dashboard!”) misses what actually happened.

Two things genuinely surprised me. First, context retention. I described the MCC hierarchy and the product segmentation once, early in the session, and that context got used throughout. When I asked “should MCC 5045 and 5065 go in the same bucket?” — 5045 is computers/peripherals, 5065 is electrical apparatus — I got back a substantive answer about the tradeoff, not a generic response. The domain framing I provided stuck.

Second, the parts I thought would be hard were fast. The data model took maybe twenty minutes total. Faker configuration, pandas wrangling, Streamlit layout, Plotly figure construction — all of this was quick. These are the parts where I used to get stuck googling syntax.

What took time were the judgment calls that were genuinely mine. Deciding the right MCC bucketing granularity. Choosing the right affinity metric. Deciding which filters to expose in the Streamlit sidebar versus hardcoding. Figuring out what story the bubble chart should tell and encoding that in the axis choices. None of that was delegatable.

The bottleneck shifted from “can I write this code” to “do I know what to build.” That’s a meaningful change if you think about where analyst time was going before.

The result

The dashboard has four panels: a spend-by-category bar chart, a category co-occurrence heatmap, the affinity bubble chart, and a filterable transaction table. The synthetic dataset runs to ~50,000 transactions across 200 companies, 12 MCC categories, and 3 product types — enough to make the visualizations feel real.

It’s not a production analytics tool. The data is fake, the company attributes are randomly generated, and the affinity metric is a first-pass proxy. But it’s a working, shareable artifact that demonstrates a real analytical idea — and it exists because I was able to ship a complete thing in a single afternoon instead of waiting on engineering.

What this means if you’re an analyst

The mental model I find useful: Claude Code is a very capable junior engineer who knows all the syntax and APIs cold, but has no domain knowledge and no opinion about what should be built. You still have to be the person who knows that virtual card transactions have fat-tailed amount distributions. You still have to know that MCC codes need bucketing to be analytically useful. You still have to decide what story the bubble chart should tell.

What you don’t have to bring: pandas syntax, Streamlit layout APIs, Plotly figure construction, Faker providers, data type handling. The mechanics of implementation can be delegated. The understanding of the problem cannot.

If you’re a product analyst who writes Python occasionally and has a half-formed idea for a dashboard or tool you’ve been putting off because it would take more eng-style work than you have capacity for — the calculus has changed. The domain expertise you’ve been building is exactly what the tool needs from you. The rest moves faster now.
